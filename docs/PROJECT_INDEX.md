## 1. プロジェクト概要

### 目的
心不全で退院した親の異変（体重・息切れ）に、離れて暮らす子が気づいて**電話をかけるきっかけを作る**、4日間スプリントの MVP プロトタイプ。

### 体制
- **チーム**: 3名（プログラミング学習中、SUNABACO在籍）
  - **A**: サーバー/DB担当
  - **B**: 親画面担当
  - **C**: 子画面担当

### 期間とゴール
- **期間**: 2026-04-27（月）〜 04-30（木）／実働4日
- **ゴール**: 4/30 終業時に身内デモ可能な動作プロトタイプ
- **総稼働**: 約37.5h（各12.5h × 3名）

### 技術スタック
- Backend: Flask / SQLAlchemy / SQLite
- Frontend: HTML / Vanilla CSS / Vanilla JS / Chart.js (CDN)
- DB Migration: Alembic
- Auth: Flask セッション + bcrypt
- TZ: Asia/Tokyo 固定

## 4. 重要な設計判断（背景理解用）

### 4.1 なぜ M:N スキーマで UI は 1:1 制約か
- 中規模到達時（〜1000家族）に1親:N子（兄弟分担）/ M親:N子（父母両方見守り）が必須
- 後付けマイグレーションは2〜3週間規模、初期から M:N なら追加コスト1日
- ただし MVP 4日間で UI まで M:N 対応は不可能 → スキーマだけ M:N、UI は 1:1 制限

### 4.2 なぜ MVP で両者ログインか
- M:N で複数の子が同じ親を見守る場合、token URL方式では「誰が既読か」追跡不能
- 医療データ漏洩リスク（LINE転送・スクショ流出）を考えるとPIN/パスワード保護必須
- 子は招待トークンで初回登録 → 以降はパスワードログイン

### 4.3 なぜメール通知を P3 撤退したか
- SMTP 設定ハマりで Day 4 が崩壊するリスクが高い
- 4日でデモ規模なら子が能動的にアプリを開く運用で代替可能
- フェーズ2（GW以降）で実装

### 4.4 なぜ alert_logic_version カラムが必要か
- 心不全アラート閾値は実運用で必ず微調整したくなる
- 過去データの判定根拠を保持するため、判定時のバージョンを記録
- カラム1個で将来の閾値変更を許容
