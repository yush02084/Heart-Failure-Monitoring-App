## 4. Git / ブランチ運用ルール（推奨）

### ブランチ戦略
```
main                  ← protected、direct push禁止
├── feature/server    ← A
├── feature/parent    ← B
└── feature/watcher   ← C
```

### Daily ワークフロー
1. 朝の sync 後、各自 main から最新を pull → 自分のfeatureブランチに rebase
2. 作業 → コミット
3. 一日の終わりに main へ PR、他メンバー1人が approve
4. 翌朝 sync 前にマージ

### コンフリクト回避
- `templates/base.html` を触るときは Slack/LINE で予告
- `app/__init__.py` の編集は A のみ（B/C はリクエスト送る）
- `requirements.txt` の追加は PR で全員に共有

### コミットメッセージ規約
```
[A] models: add Watch_Relationships table
[B] parent: implement daily input form
[C] watcher: add tel button to detail page
[共通] base.html: add viewport meta tag
```

## 6. API契約（Day 1 朝に確定すべき内容）

### ルート一覧（SSR、HTMLを返す）

| メソッド | パス | 担当 | 役割 |
|---|---|---|---|
| GET | `/` | A | ランディング → 認証状態でリダイレクト |
| GET, POST | `/auth/register/parent` | A→B | 親の新規登録 |
| GET, POST | `/auth/login` | A→B,C | 共通ログイン |
| POST | `/auth/logout` | A | ログアウト |
| GET | `/parent/home` | A→B | 親のホーム画面 |
| GET, POST | `/parent/input` | A→B | 日次入力 |
| GET, POST | `/parent/invitations` | A→B | 招待リンク発行 |
| GET, POST | `/auth/register/watcher/<token>` | A→C | 子のトークン経由登録 |
| GET | `/watcher/dashboard` | A→C | 子のダッシュボード |
| GET | `/watcher/parent/<int:parent_id>` | A→C | 親詳細 |

### コンテキスト変数（Flask render_template に渡す辞書）
- 各ルートで何を渡すかを A が `routes.py` のコメントで明示
- 例：`/parent/home` は `{ user, today_record, recent_records (list of 7) }` を渡す
- B/C はそれを前提にテンプレート設計

## 7. リスクと対応

| リスク | 兆候 | 対応 |
|---|---|---|
| Day 1 で API契約が決まらない | sync が長引く | A 主導で「とりあえず案」で進める、Day 2 朝に微修正可 |
| B/C が A のモデル完成を待ってブロック | Day 1 後半で詰まる | B/C は Day 1 はモック JSON でフロント先行、Day 2 で結合 |
| Git コンフリクト多発 | merge ハマる | base.html / __init__.py の編集予告ルール厳守 |
| 1人だけ休み | 当日0時間メンバー | 残り2人で当該タスクを巻き取り、優先度低タスクを切る |
| Day 3 結合で大量バグ | 統合できない | Day 3 後半1hを統合専用に確保、P1 全切り |
| メンバー間スキル差 | 進捗ばらつき | 上位者が tasks/ ディレクトリ内の代表実装を1つ書いて参考にする |
