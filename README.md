# Heart-Failure-Monitoring-App
心不全患者記録共有型の見守りアプリ
> 離れて暮らす家族が「親が今日も大丈夫」を知るための、心不全特化の見守りWebアプリ

SUNABACO PBL 卒業制作

---

## ⚠️ 重要：医療免責事項

- 本アプリは**プロトタイプ**です。医療機器ではありません
- 症状の診断や治療判断を行うものではなく、受診の意思決定は必ず医師が行います
- 実運用を想定したものではなく、**模擬データでの動作確認のみ**を目的としています
- 実際に症状が気になる場合は、かかりつけ医を受診してください

---

## 🎯 なぜ作ったか

心不全患者は国内に約120万人、今後さらに増加する「心不全パンデミック」が課題とされています。
退院後30日は再入院リスクのピーク期ですが、離れて暮らす家族の多くは「今日の親が大丈夫か」を知る手段を持ちません。

既存の体調管理アプリは完成度が高いものの、本人の自己管理が主語で、離れて暮らす家族が主体的に関わる設計にはなっていません。

我々はターゲットを反転させ、**子世代を主語**にした見守りツールとして再設計しました。

- **親**は10秒、体重と息切れを入れるだけ
- **子**は毎朝、スマホで「今日の親」を1画面で確認
- **異変時**は子のスマホにアラート、親に電話するきっかけを作る

---

## 🧭 設計思想

| 軸 | 方針 |
|---|---|
| 主顧客 | 離れて暮らす子世代（40〜60代） |
| 協力者 | 心不全で退院した高齢の親 |
| 記録項目 | 体重・息切れの2つのみ（最小入力） |
| 判定 | 黄信号／赤信号の2段階、アラートは家族目線の文言 |
| トーン | 医師の指示ではなく、家族から家族への促し |

**やらないこと**：
- 医療従事者向けの機能
- 詳細すぎる症状入力や複雑な記録
- 患者の行動（入力・受診）を強制する仕組み
- スマートフォンを利用できない人への対応
- 医療機器レベルの厳密な数値管理
- 医療判断（診断・治療の代替）
- 多機能化は行わない（機能は最小限に絞る）
- デザイン性を優先したスタイリッシュなUI

## ✨ 機能

### 親画面（入力）

- ログイン画面
- 体重入力
- 息切れ入力

### 子画面（ダッシュボード）

- 共有URL（`/family/<token>`）でログイン不要アクセス
- 「今日の親」状態カード（🟢🟡🔴）＋家族目線のメッセージ
- 過去7日の体重推移グラフ（Chart.js）
- 息切れ履歴リスト、アラート履歴

### 判定ロジック

| レベル | 条件（OR） |
|---|---|
| 🔴 critical | 息切れ「とても／けっこう苦しい」／体重3日で +2.0kg 以上 |
| 🟡 warning | 息切れ「ちょっと苦しい」／体重3日で +1.5〜+1.9kg |
| 🟢 safe | 上記以外 |

記録1〜3日目は基準体重（退院時体重）との差、4日目以降は3日前との差で判定します。

## 🗄 データモデル（概要）

### parents
親アカウント情報（ログインID・PINハッシュ・親子の呼称・基準体重・家族共有用トークン）

### daily_records
日次記録（体重・息切れ・判定結果・子の既読フラグ）。`(parent_id, record_date)` のUNIQUE制約により同日複数入力はUPSERT。

---

## 🧪 動作確認シナリオ

1. `/setup` で親アカウントを作成
2. 登録後に表示される「家族共有URL」を子役のスマホに送信
3. 親端末で体重・息切れを入力
4. 子端末で共有URLにアクセス、状態カードを確認
5. 体重を +2.0kg にして再入力 → 子画面で 🔴 critical が出ること

---

## 🔒 セキュリティ上の制約（プロトタイプ）

- 共有URLはトークン付きだがログイン不要設計（URLが漏れると閲覧可）
- CSRF 対策は省略
- PIN は 4桁（10,000通り）、ブルートフォース対策なし
- **以上の理由から、本番運用には追加実装が必要です**
